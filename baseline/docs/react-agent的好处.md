# react agent 的实践的场景：

1. 收集一个行业赛道的分析结果


- 避免了一次性搜集全部信息导致的局部信息不准确
    
- 需要中间数据来验证思考，而非凭空靠直觉决策

```text
Think-1：判断赛道潜力，需要 “政策支持力度、行业增速、龙头公司盈利能力、产业链瓶颈”4 类信息。
Act-1：调用 API 获取行业财报整体数据
Think-2：分析数据，判断行业高增长 + 政策背书，但上游价格上涨可能挤压中下游利润，需要进一步验证是否有影响
Act-2: 调用 API 获取供需、行业研报等详细数据
Think-3: 整合结论生成分析报告，附关键数据来源
```

2. 一个问题很难解决，即需要多步的复杂推理。

- 逐步缩小问题范围，避免盲目操作
- 每一步都有依据，便于后续训模型和验证链路

```text
Think-1：理清故障的常见原因，例如宕机的常见原因是 “CPU 过载、内存不足、磁盘满、服务崩溃”，需要先查基础监控数据
Act-1：调用「监控系统 API」查询服务器打点数据
Think-2：判断主因，例如 CPU 利用率异常则进一步排查哪些进程 CPU 占用高
Act-2：用「进程管理工具」查 TOP 进程，看是否有异常服务
Think-3：发现日志服务异常，可能是 “日志文件过大” 或 “配置错误”，需要进一步查看日志服务的配置和日志文件大小
Act-3：bash 执行命令，发现日志文件过大，同时配置未开启滚动，也未设置最大日志大小
Think-4：向运维工程师提供可行的解决方案：清理日志，修改配置并开启滚动，重启日志服务与应用
```

# 与COT模式的区别

COT模式是在prompt里面显示的写出：请一步一步的推理，然后再把答案放在哪个标签里面。（一般是用一个思考模型，让其在think内进行逐步推理，来验证结果的可靠性）

但是cot模式下，如果模型需要一个数据验证，这个时候它本身是做不到的。

# 与func calling模式的区别

func calling工作流程：

```text
┌─────────────────────────────────────────────────────────────────┐
│                    Function Calling 工作流程                      │
└─────────────────────────────────────────────────────────────────┘

用户: "北京今天天气怎么样？" (然后在system prompt里面注入对应的tool call,告诉LLM你可以调用什么工具)
         │
         ▼
┌─────────────────┐
│   LLM (Agent)   │
│ 分析用户意图     │
│ 决定调用函数     │
└────────┬────────┘
         │ 输出: {"name": "get_weather", "arguments": {"city": "北京"}}
         ▼
┌─────────────────┐
│   函数执行器     │
│ 调用实际函数     │
│ 获取天气数据     │
└────────┬────────┘
         │ 返回: "晴天 25°C"
         ▼
┌─────────────────┐
│   LLM (Agent)   │
│ 基于结果生成回复 │
└────────┬────────┘
         │
         ▼
用户: "北京今天晴天，气温25°C"
```

这本质上就是react的其中一步

# react的结果收敛问题

## 1. plan尝试

用一个plan作为全局的规划器，就是不让react走一步看一步，这样可以防止一步错步步错。然后在特定的步长的时候，让plan更新。

## 2. 问题重申

每轮思考的时候，都重申一遍原始问题，这样防止跑题

## 3. 早停

重复调用一个工具K次以上，强制停止流程。让LLM基于之前的链路给出答案

## 4. SkillRL

有一篇论文，SkillRL里面有一个思想：通过长期反思（总结失败原因并存入长期记忆）和状态对齐（持续检查当前状态与目标的差距）来指导后续行动。
